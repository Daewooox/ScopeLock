# Когда класть документ в `backlog/`

В `backlog/` — **парковка отложенной работы**: то, что уже обдумано/обсуждено/частично исследовано, но сознательно отложено «на потом». Не «идея на подумать», а конкретный пункт работы с достаточным контекстом, чтобы будущий ты (или другой агент) поднял его и не начинал с нуля.

## Подходит

- Задача, которую решили отложить (не приоритет сейчас), но жалко терять контекст исследования.
- Фолоу-апы из завершённой задачи («вот это ещё стоило бы сделать»).
- Техдолг / рефакторинг, который вызрел, но ждёт своего окна.
- Крупная миграция/фича, которую надо сначала обдумать, прежде чем заводить через `/van`.

## Не подходит

- Активная задача в работе — это `tasks.md`.
- Детальный план уже взятой в работу задачи — это `plans/`.
- Завершённая работа — это `archive/`.

## Имя файла

- `backlog-<slug>.md` — `backlog-overscroll-behavior.md`, `backlog-react-19-upgrade.md`
- Если пункт вырос из конкретной задачи — `backlog-NNNN-<slug>.md` — `backlog-0372-followups.md`

## Структура (свободная, но с контекстом)

Минимум — чтобы будущий решающий понял, не поднимая всю историю:

```markdown
# Беклог — <название>

**Статус:** отложено / подумать / готов к /van
**Создано:** YYYY-MM-DD
**Контекст обсуждения:** задача #NNNN (откуда выросло)

## Суть
Что и зачем.

## Почему отложили
Чтобы через месяц не гадать «а чего не сделали-то».

## Возможные направления
1. Вариант A — ...
2. Вариант B — ...

## Что важно учесть при принятии решения
- подводные камни, ограничения.

## Связанные файлы
- `src/...` — точки приложения правки.

## TL;DR будущему решающему
Короткий вывод: если делать — делать так-то.
```

## Жизненный цикл

- Отложили работу → `backlog/backlog-<slug>.md`.
- Решили взять в работу → заводи через `/van` (backlog-файл = готовый контекст для VAN/PLAN). После старта файл можно оставить как референс или удалить.
- Передумали → пометь `**Статус:** rejected — <причина>` или удали.
