# Когда класть документ в `audit-sql/`

В `audit-sql/` — **SQL-скрипты для предварительной проверки** перед миграцией данных или схемы. Дешёвый способ узнать реальное состояние БД до того, как принять решение об уровне сложности задачи.

## Подходит

- `SELECT` для подсчёта коллизий по новому уникальному ключу.
- `SELECT` для распределения значений в поле (сколько NULL, сколько уникальных).
- `SELECT` для подсчёта строк, которые попадут под `UPDATE` миграции.
- Поиск JSONB-ссылок на изменяемые ID.
- Любая «разведка» по БД, которая меняет уровень сложности задачи на 1-2 ступени.

## Не подходит

- Сама миграция — это `migrations/` или `supabase/migrations/`, не сюда.
- Production-патчи — это часть кода, а не аудит.
- One-shot ad-hoc SELECT в Studio — не нужно ничего комитить.

## Имя файла

- `NNNN-<topic>.sql` — `0362-group-chat-media.sql`, `0364-media-links-tab.sql`
- Опционально с описанием в комментариях вверху файла.

## Почему это важно

Этот паттерн (audit-SQL до VAN/PLAN) — фиксированный урок: цифры из БД могут изменить уровень сложности задачи на 1-2 ступени. Дешевле потратить 30 секунд на 3 SELECT'а, чем потом часами отлаживать миграцию с непредвиденными конфликтами.

Применимо не только к SQL — но и к `curl` для внешних API, `node -e` для парсеров URL и i18n путей.
