# Replace the hand-drawn hero SVGs with real VHS terminal recordings

## Problem

`README.md` embeds two animated SVGs (`docs/assets/scopelock-demo.svg`,
`docs/assets/scopelock-plan-demo.svg`) generated by `scripts/demo-svg/`
(PR #65). That pipeline already runs the real CLI and captures real output,
but renders it inside a hand-drawn SVG "terminal chrome" (a programmatically
laid-out `<rect>`/`<text>` box with CSS-keyframe reveal) rather than an
actual terminal. The user's assessment, verbatim: the animations "look
somewhat artificial" ("выглядят как-то искусственно"). The custom chrome
does not match any real terminal's font metrics or rendering, and the
reveal timing is mechanically uniform rather than a captured pace.

There is also no artifact usable for promoting the project outside the
README (Twitter/X, LinkedIn) - the SVGs are only useful embedded inline.

## Goals

- Replace both hero animations with recordings of a real terminal (via
  [VHS](https://github.com/charmbracelet/vhs)) running the same two
  scenarios as today, unchanged in content: Guided (`task start` +
  `task finish`) and Standard (`plan prepare` with a writer/reader
  read-hazard pair).
- Each recording is driven by a checked-in `.tape` script that types the
  real `scopelock` command line into a real shell - VHS records an actual
  terminal, not a replay of pre-captured text.
- Two outputs per scenario from one recording: a GIF (committed, embedded
  in `README.md` exactly as the SVGs are today) and an MP4 (generated
  on demand for external sharing, never committed).
- A CI job that fails when the `.tape` scripts' commands no longer run
  cleanly against the current CLI, so a silent regression (renamed flag,
  broken command) cannot ship unnoticed. This job does not compare
  rendered video output - see Non-goals.
- Same fixture-repository setup already proven correct in
  `scripts/demo-svg/capture.mjs` (git init, fake `codex` executable,
  `approveContract` helper) - reused, not reimplemented.

## Non-goals

- No byte-identical / pixel-identical CI staleness check on the rendered
  GIF/MP4. VHS renders through a real terminal (ttyd + headless Chromium),
  so exact reproducibility across CI runs is not guaranteed the way pure
  SVG text generation was. Regenerating and reviewing the video by eye,
  by hand, remains a manual step after a CLI-output-affecting change.
  CI instead re-runs the exact commands the `.tape` scripts type and
  asserts they still succeed - this catches "the demo would now be wrong"
  without needing a video diff.
- No change to demo *content* - same two scenarios, same commands, same
  fixture setup logic as the existing SVG pipeline. This is a rendering
  swap, not a demo redesign.
- No WebM output. MP4 already covers the "share outside GitHub" need;
  WebM adds a third artifact with no platform that needs it here.
- No commit of the MP4 to the repository. It's an external marketing
  artifact, not something the project needs at rest; committing binary
  video would only bloat repository history across every regeneration.
- Does not touch `benchmarks/coordination/run-*-demo.mjs` (different
  purpose, `--json`-mode evidence generation, already out of scope for
  the SVG pipeline too).

## Architecture

### Fixture setup: `scripts/demo-vhs/setup-fixture.mjs <guided|plan>`

A thin Node CLI wrapper around the exact functions `scripts/demo-svg/
capture.mjs` already exports (`initFixtureRepo`, `fakeCodexOnPath`,
`approveContract`). Given a scenario name, it:

- creates a throwaway git fixture repo (via `initFixtureRepo`),
- writes a `scopelock` shim into the same fixture `.demo-fake-bin`
  directory `fakeCodexOnPath` already creates - a one-line executable that
  `exec`s `node <repoRoot>/packages/cli/dist/index.js "$@"` - so one `PATH`
  prepend in the tape's setup block resolves both `scopelock` and (for
  `guided`) the fake `codex`,
- for `guided`: writes the fake `codex` executable on `PATH` and commits
  it as part of the baseline (same ordering `generate.mjs` uses today, so
  it never shows up as drift),
- for `plan`: approves the `writer` (`src/writer.js`) and `reader`
  (`src/reader.js`, reads `src/writer.js`) contracts and writes
  `plan.json`,
- prints the fixture directory's absolute path to stdout (nothing else),
  so the calling `.tape` script's `Hidden` block can `cd` into it.

`capture.mjs` and `sanitize.mjs`'s SVG-specific pieces (recordingReporter,
sanitizeHuman's placeholder substitution) are not needed here - VHS
records the real terminal, so there is no captured-text string to sanitize
and no synthetic reveal timeline to build from `ProgressEvent`s.

### `.tape` scripts

`scripts/demo-vhs/guided.tape` and `scripts/demo-vhs/plan.tape`. Shape
(guided shown, plan is the same pattern with its own commands):

```
Output docs/assets/scopelock-demo.gif
Output .demo-vhs-out/scopelock-demo.mp4
Set FontFamily "..."
Set Width 960
Set Height 360
Set Theme "..."

Hidden
Type "cd $(node scripts/demo-vhs/setup-fixture.mjs guided) && export PATH=\"$(pwd)/.demo-fake-bin:$PATH\" && clear"
Enter
Show
```

(`setup-fixture.mjs` prints the fixture directory path; `.demo-fake-bin`
inside it holds both the `scopelock` shim and, for `guided`, the fake
`codex` executable - see below.)

```

Type `scopelock task start "Add a dark mode toggle" --agent codex --allow src --yes`
Enter
Sleep 500ms

Hidden
Type "mkdir -p src && printf 'export const darkMode = true;\n' > src/dark-mode.js && printf 'import { test } from \"node:test\";\nimport assert from \"node:assert/strict\";\nimport { darkMode } from \"./dark-mode.js\";\n\ntest(\"dark mode is enabled\", () => { assert.equal(darkMode, true); });\n' > src/dark-mode.test.js && clear"
Enter
Show

Type "scopelock task finish"
Enter
Sleep 1s
```

Exact `Set` values (font, theme, window chrome) are an implementation
detail decided during implementation to visually match the current SVGs'
dark palette as closely as VHS's config allows - not spec-critical.

`scopelock` on `PATH` resolves to the repo's own built CLI: the setup
`Hidden` block prepends the fixture's fake-codex bin dir and the real
`packages/cli/dist` bin (via a local `PATH` export, mirroring how
`pnpm --filter @scopelock/cli link --global` would work) rather than a
global install, so the recording always uses the just-built CLI.

The plan scenario's `--validation-command` uses the same placeholder as
today: `node -e "process.exit(0)"` (never `process.execPath`, to avoid
baking a machine-specific absolute path into the recording - the same
reason `generate.mjs` avoids it today).

### `pnpm demo:vhs`

```json
"demo:vhs": "pnpm build && vhs scripts/demo-vhs/guided.tape && vhs scripts/demo-vhs/plan.tape"
```

Requires `vhs` on `PATH` locally (`brew install vhs` on macOS, which pulls
in `ttyd`/`ffmpeg`). Running it regenerates both GIFs in `docs/assets/`
and both MP4s in a new `.demo-vhs-out/` (gitignored).

### CI: `demo-vhs-check` job

Replaces `demo-svg-check` in the release-readiness/test workflow(s).
Ubuntu-only (matches today), using `charmbracelet/vhs-action` to install
`vhs`/`ttyd`/`ffmpeg` rather than hand-rolled apt steps. The job does not
invoke `vhs` at all - it runs `node scripts/demo-vhs/smoke.mjs`, which:

- reads each `.tape` file,
- extracts every visible `Type` line's literal command text,
- replays those exact command lines against a fixture built the same way
  `setup-fixture.mjs` builds it, asserting each exits `0`.

This catches the failure mode that actually matters (a documented demo
command now errors or behaves differently) without asserting anything
about pixels, fonts, or timing.

### Deleted

- `scripts/demo-svg/` in full: `capture.mjs`, `sanitize.mjs`, `render.mjs`,
  `generate.mjs`, and their four `*.test.mjs` files.
- `docs/assets/scopelock-demo.svg`, `docs/assets/scopelock-plan-demo.svg`.
- `pnpm demo:svg` / `pnpm demo:svg:check` scripts.
- `demo-svg-check` CI job.

### `README.md`

Both `<img src="./docs/assets/scopelock-demo.svg" ...>` /
`scopelock-plan-demo.svg` become `.gif`. The line "The animation above is
generated from real `scopelock` command output - regenerate it with `pnpm
demo:svg`" becomes "...with `pnpm demo:vhs`". The reduced-motion note
("With reduced-motion enabled it stays on the completed frame") no longer
applies - GIF has no `prefers-reduced-motion` hook the way inline SVG/CSS
did - so that sentence is removed rather than rewritten inaccurately.

## Testing

- `scripts/demo-vhs/setup-fixture.mjs` gets the same kind of unit coverage
  `capture.mjs` has today (fixture repo is valid git, fake codex resolves,
  contracts approve cleanly) - ported/adapted, not newly invented.
- `scripts/demo-vhs/smoke.mjs` is exercised directly in CI (that is its
  test) and additionally covered by a unit test asserting it correctly
  extracts `Type` lines from a sample `.tape` fixture string.
- Manual verification step (documented in the implementation plan, not
  automated): after implementation, run `pnpm demo:vhs` locally, open both
  GIFs, and visually confirm parity of content with the removed SVGs
  before committing them.
